# Inventory Tracker Devlog — 제품 보관·복원과 안전 삭제

- **Date**: 2026-07-21
- **Repository**: `mworkroom/inventory-tracker`
- **PR**: #8
- **Merge commit**: `0f39f4c`

## 배경

제품 설정에 삭제 버튼이 없고, 개수형·용량형도 등록 후 변경할 수 없게 되어 있는 이유를 검토했다.

제품은 다음 기록의 부모이므로 일반적인 hard delete를 허용하면 과거 데이터가 사라지거나 관계가 깨질 수 있었다.

```text
inventory_events
inventory_usage_cycles
inventory_purchases
```

따라서 제품 관리 방식을 다음처럼 결정했다.

```text
제품 설정
├─ 제품 보관
│  └─ 목록에서 숨김, 기록 유지, 언제든 복원
│
└─ 잘못 만든 제품 삭제
   └─ 실사용 기록이 전혀 없을 때만 가능
```

## 구현 내용

### 제품 보관

- 제품 설정 화면에 `제품 보관` 추가
- 보관 시 `is_archived = true`로 변경
- 기본 재고 목록과 구매처별 목록에서 숨김
- 재고 이벤트, 사용 주기, 구매 기록은 모두 유지
- 현재 개봉하여 사용 중인 제품은 소진 전 보관 불가
- 실행 전 버튼을 한 번 더 눌러 확인

### 보관된 제품 화면

설정 메뉴에 `보관된 제품` 항목을 추가했다.

- 보관된 제품 수 표시
- 제품명, 현재 재고, 주구매처 표시
- 제품별 복원 버튼
- 복원하면 기존 기록을 유지한 채 기본 목록으로 돌아옴
- 같은 이름의 활성 제품이 있으면 복원 차단

### 잘못 만든 제품 삭제

다음 조건을 모두 만족할 때만 영구 삭제 버튼이 활성화된다.

```text
active_opened_on 없음
inventory_events가 정확히 1건
해당 이벤트가 자동 생성된 최초 재고 등록 이벤트
inventory_usage_cycles 없음
inventory_purchases 없음
```

삭제 시:

1. 자동 생성된 최초 재고 이벤트 삭제
2. 제품 행 삭제

두 작업을 하나의 Supabase 함수 안에서 실행한다.

실사용 또는 구매 기록이 하나라도 있으면 삭제하지 않고 제품 보관을 안내한다.

## Supabase RPC

```text
set_inventory_product_archived(product_id, archived)
delete_unused_inventory_product(product_id)
```

`delete_unused_inventory_product()`는 `SECURITY DEFINER`로 실행되지만 다음을 직접 검증한다.

- 현재 사용자가 해당 workspace 구성원인지
- 실사용 이벤트가 없는지
- 완료된 사용 주기가 없는지
- 구매 기록이 없는지

브라우저 UI를 우회해 RPC를 직접 호출해도 기록이 있는 제품은 삭제할 수 없다.

## UI 안전장치

- 보관 버튼 2단계 확인
- 영구 삭제 버튼 2단계 확인
- 삭제 불가능한 경우 버튼 비활성화 및 이유 표시
- 사용 중 제품의 보관 버튼 비활성화
- 보관과 삭제 오류를 제품 설정 화면에 표시

## 검증

### 보관·복원·삭제

테스트 제품을 생성한 뒤 다음 순서를 검증했다.

```text
제품 생성
→ 보관
→ 복원
→ 영구 삭제
```

삭제 후 제품 행과 최초 재고 이벤트가 모두 제거된 것을 확인했다.

### 기록이 있는 제품 삭제 차단

테스트 제품에 구매 기록을 추가한 뒤 삭제 RPC를 호출했다.

결과:

```text
삭제 차단: 성공
제품 보존: 성공
오류 문구:
실사용 또는 구매 기록이 있는 제품은 삭제할 수 없습니다. 제품 보관을 사용해주세요.
```

## 현재 제품 관리 원칙

```text
더 이상 쓰지 않는 제품
→ 제품 보관

실수로 방금 잘못 만든 제품
→ 잘못 만든 제품 삭제

사용·구매 기록이 쌓인 제품
→ 영구 삭제 불가, 제품 보관만 가능
```

개수형·용량형 변경 제한은 유지한다. 과거 수량과 이벤트의 단위 해석이 바뀌어 계산이 손상될 수 있기 때문이다.
