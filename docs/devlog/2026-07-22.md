# 2026-07-22 Inventory Tracker Development Log

## Work Summary

제품 항목을 만드는 순간 현재 재고까지 확정해야 했던 흐름을 분리했다. 실제 사용 방식과 맞지 않고 입력 부담이 큰 `용량 직접 차감`은 제거하고, 새 제품은 기본 설정만 저장한 뒤 첫 입고 또는 현재 재고 설정 시점부터 재고 계산을 시작하도록 정리했다.

## Work Completed

- 새 제품의 기록 방식을 `개수 직접 차감`, `개봉 → 소진` 두 가지로 축소했다.
- 신규 제품에서 현재 재고와 최초 재고 확인일 입력을 제거했다.
- `stock_initialized` 상태를 추가하는 Supabase 마이그레이션을 작성했다.
- 재고 미설정 제품은 0개 또는 품절로 해석하지 않고 회색 상태와 `재고 미설정` 문구를 표시하도록 변경했다.
- 첫 입고는 0에서 시작하는 첫 재고 기준점이 되고, 현재 재고 설정은 이미 보유한 수량을 기준점으로 저장하도록 재고 기록 함수를 변경했다.
- 재고 미설정 상태에서는 사용·개봉을 막고, 구매 기록 입력은 계속 허용했다.
- 실제 사용 주기가 부족하면 과거 구매 간격을 `구매 기록 기준 임시 예상`으로 사용하고, 개봉·소진 또는 사용 기록으로 계산 가능한 시점부터 실제 사용 예측을 우선하도록 했다.
- 구매 필요 상태는 현재 재고 개수 또는 예상 잔여일 기준으로 판단하도록 유지하고, 포장형 제품의 g·ml는 현재 제품 잔량과 주기 보정에만 사용하도록 문구를 정리했다.
- 재고 기준점이 없는 제품과 구매 기록 임시 예측을 검증하는 테스트를 추가하고, 용량 직접 차감 테스트를 제거했다.
- `npm.cmd run verify`로 Vite 프로덕션 빌드와 11개 계산 테스트 통과를 확인했다.

## Decisions

- `모르는 재고`와 `실제 0개`는 다른 상태이므로 숫자 0 하나로 표현하지 않고 별도 초기화 상태로 관리한다.
- 제품 생성은 품목과 계산 규칙을 등록하는 작업으로 한정하고, 현재 재고 기준점은 나중에 연결할 수 있게 한다.
- 첫 입고는 이전 재고가 0이었다는 의미가 분명하므로 별도 재고 설정 없이 계산을 시작할 수 있다.
- 과거 구매 간격은 할인·묶음 구매의 영향을 받을 수 있으므로 실제 소진일로 단정하지 않고 사용 기록이 쌓이기 전의 임시 참고값으로만 사용한다.
- 기존 운영 데이터에는 `capacity` 제품이 없음을 읽기 전용으로 확인했으므로 데이터 변환이나 삭제 없이 허용 방식만 축소한다. 마이그레이션도 `capacity` 행이 발견되면 중단하도록 방어 조건을 두었다.
- 프런트엔드와 DB의 적용 순서가 잠시 어긋나도 기존 제품을 재고 미설정으로 오해하지 않도록 이전 스키마 호환 처리를 유지한다.

## Current Status

### Work Completed

- 제품 카드의 `사용 주기 기록` 영역에 `과거 기록 추가` 버튼과 입력창을 추가했다.
- 과거 기록은 개봉일, 다 쓴 날, 당시 사용 인원을 받으며 사용 인원 기본값은 1명이다.

## 사용 중 정보와 사용 주기 정정 기능

- 수정 시 `inventory_products`의 현재 사용 정보와 연결된 `open` 이벤트가 한 트랜잭션에서 함께 바뀌는 `update_active_usage` RPC를 추가했다.

- 운영 DB에 `safe_record_corrections` 마이그레이션을 적용하고 RPC 권한, RLS 정책, 컬럼별 권한과 기간 계산 트리거를 확인했다.
- 코코넛 오일의 개봉 기록 2명에 맞춰 `current_consumer_count`와 `active_consumer_count`를 모두 2명으로 정정했다. 재고 1개, 개봉일 2026-05-01, 잔량 1600ml는 변경하지 않았다.
- `npm.cmd run verify`에서 프로덕션 빌드와 계산 테스트 12개가 모두 통과했다.

### Decisions

- 과거 재고 수량 이벤트를 직접 덮어쓰거나 삭제하면 현재 재고와 전후 수량 연결이 어긋날 수 있으므로 수량 오류는 계속 `재고 정정` 이벤트로 처리한다.
- 사용 인원과 사용 기간은 수량 원장을 바꾸지 않으면서 예측 결과에는 영향을 주므로 별도의 안전한 수정 경로를 제공한다.
- 현재 제품의 사용 인원은 제품 설정값만 바꾸지 않고, 활성 사용 정보와 연결된 개봉 기록까지 함께 수정해야 소진 시 올바른 사용 주기가 생성된다.
- 완료된 사용 주기 삭제는 재고 수량에 영향을 주지 않고 예측 학습 자료만 제거하므로 확인 절차를 거쳐 허용한다.
- 새 RPC는 `security invoker`로 실행하고 익명 호출을 막았으며, 기존 workspace 멤버 RLS를 그대로 적용한다.

### Current Status

- 프런트엔드 코드와 운영 Supabase 스키마 적용이 완료됐다.
- 코코넛 오일은 현재 사용 정보와 개봉 기록이 모두 2명으로 일치한다.
- 빌드와 12개 계산 테스트, 운영 DB의 함수 권한·RLS·트리거를 확인했다.
- Security/Performance Advisor에는 이번 변경으로 새로 생긴 경고가 없고 기존 다른 앱 및 기존 함수 관련 경고만 남아 있다.
- 변경 코드는 아직 커밋하거나 GitHub에 푸시하지 않았다.

### Next Steps

- 로그인된 앱에서 `사용 중 수정`과 완료된 사용 주기 수정·삭제 화면을 실제 데이터로 한 번 확인한다.
- 확인 후 GitHub Issue #9와 연결한 커밋으로 `main`에 반영한다.

## 카테고리와 그룹 보기 추가

### Work Summary

제품을 카테고리로 분류하고, 메인 화면에서 구매처 또는 카테고리 기준으로 제품을 묶어 볼 수 있게 했다. 카드 통계의 핵심 정보 순서도 사용자가 요청한 흐름에 맞게 조정했다.

### Work Completed

- `inventory_products.category` 컬럼과 기존 제품용 기본값 `미분류`를 추가하는 Supabase migration 작성
- 제품 추가·수정 화면에 카테고리 입력창 추가
- 카드 통계 상단을 `카테고리 → 주구매처 → 현재 재고 → 제품 1개 용량 → 기록 방식 → 현재 제품 → 학습한 주기` 순서로 정리
- 메인 보기 전환을 `구매처 / 카테고리`로 변경
- 카테고리별 제품 그룹, 제품 수, 구매 필요 수, 30일 안 구매 예상 수 표시
- 검색어에 카테고리도 포함
- 기존 테스트 fixture와 migration 전 응답을 위해 제품 category 읽기 fallback 유지

### Decisions

기존 제품은 카테고리를 임의로 추정하지 않고 `미분류`로 채운다. 카테고리는 제품명과 별개의 사용자 관리 값이므로 자동 분류하지 않고, 제품 설정에서 직접 수정하도록 했다.

### Current Status

프런트엔드 코드와 migration 작성이 완료됐고 Vite 빌드와 계산 테스트 12개가 통과했다. 운영 Supabase에는 migration을 아직 적용하지 않았으며, 배포 전 migration 적용과 로그인 후 실제 그룹 보기 확인이 필요하다.

### Next Steps

- 운영 Supabase에 카테고리 migration 적용
- 제품 설정에서 기존 제품 카테고리 입력
- 배포 후 구매처·카테고리 그룹 보기와 카드 통계 순서 확인

## 고정 카테고리와 운영 스키마 적용

### Work Summary

카테고리 입력값을 실제 재고 분류에 사용할 다섯 가지로 확정하고, 제품 편집 화면과 Supabase 스키마가 같은 허용값을 사용하도록 정리했다.

### Work Completed

- 카테고리 선택지를 `식료품`, `생활용품`, `영양제`, `의복`, `미분류`로 고정했다.
- 제품 편집 화면의 자유 입력을 드롭다운으로 변경했다.
- 기존 임의 카테고리 값은 `미분류`로 정규화하도록 저장 로직과 마이그레이션을 보완했다.
- 별도 카테고리 테이블은 만들지 않고 `inventory_products.category` 컬럼의 `NOT NULL`, 기본값, 허용값 CHECK 제약으로 목록을 단일 관리한다.
- 원격 Supabase 프로젝트에 카테고리 컬럼과 허용값 제약을 적용하고 SQL 조회로 컬럼·기본값·제약을 확인했다.
- `npm.cmd run verify`와 `git diff --check`를 실행했다.

### Decisions

카테고리 목록이 사용자별로 추가·삭제되는 구조가 아니므로 별도 테이블보다 제품 컬럼의 제약과 프론트 드롭다운을 동일한 고정 목록으로 유지하는 편이 단순하고 안전하다. 기존 값이 목록 밖에 있으면 화면과 DB 제약이 충돌하지 않도록 `미분류`로 통일한다.

### Current Status

코드와 원격 Supabase 스키마가 모두 고정 카테고리 목록을 사용한다. 원격 검증 결과 `category`는 `text NOT NULL`, 기본값은 `미분류`, 허용값 CHECK 제약은 다섯 항목으로 확인됐다.

### Next Steps

- 배포 후 제품 편집 화면에서 카테고리 선택과 카테고리별 그룹 보기를 실제 계정으로 확인한다.
