# 2026-07-22 Inventory Tracker Development Log

## Work Summary

제품 항목을 만드는 순간 현재 재고까지 확정해야 했던 흐름을 분리했다. 실제 사용 방식과 맞지 않고 입력 부담이 큰 `용량 직접 차감`은 제거하고, 새 제품은 기본 설정만 저장한 뒤 첫 입고 또는 현재 재고 설정 시점부터 재고 계산을 시작하도록 정리했다.

## Work Completed

- 새 제품의 기록 방식을 `개수 직접 차감`, `개봉 → 소진` 두 가지로 축소했다.
- 신규 제품에서 현재 재고와 최초 재고 확인일 입력을 제거했다.
- `stock_initialized` 상태를 추가하는 Supabase 마이그레이션을 작성했다.
- 재고 미설정 제품은 0개 또는 품절로 해석하지 않고 회색 상태와 `재고 미설정` 문구를 표시하도록 변경했다.
- 첫 입고는 0에서 시작하는 첫 재고 기준점이 되고, 현재 재고 설정은 이미 보유한 수량을 기준점으로 저장하도록 재고 기록 함수를 변경했다.
- 재고 미설정 상태에서는 사용·개봉을 막고, 구매 기록 입력은 계속 허용했다.
- 실제 사용 주기가 부족하면 과거 구매 간격을 `구매 기록 기준 임시 예상`으로 사용하고, 개봉·소진 또는 사용 기록으로 계산 가능한 시점부터 실제 사용 예측을 우선하도록 했다.
- 구매 필요 상태는 현재 재고 개수 또는 예상 잔여일 기준으로 판단하도록 유지하고, 포장형 제품의 g·ml는 현재 제품 잔량과 주기 보정에만 사용하도록 문구를 정리했다.
- 재고 기준점이 없는 제품과 구매 기록 임시 예측을 검증하는 테스트를 추가하고, 용량 직접 차감 테스트를 제거했다.
- `npm.cmd run verify`로 Vite 프로덕션 빌드와 11개 계산 테스트 통과를 확인했다.

## Decisions

- `모르는 재고`와 `실제 0개`는 다른 상태이므로 숫자 0 하나로 표현하지 않고 별도 초기화 상태로 관리한다.
- 제품 생성은 품목과 계산 규칙을 등록하는 작업으로 한정하고, 현재 재고 기준점은 나중에 연결할 수 있게 한다.
- 첫 입고는 이전 재고가 0이었다는 의미가 분명하므로 별도 재고 설정 없이 계산을 시작할 수 있다.
- 과거 구매 간격은 할인·묶음 구매의 영향을 받을 수 있으므로 실제 소진일로 단정하지 않고 사용 기록이 쌓이기 전의 임시 참고값으로만 사용한다.
- 기존 운영 데이터에는 `capacity` 제품이 없음을 읽기 전용으로 확인했으므로 데이터 변환이나 삭제 없이 허용 방식만 축소한다. 마이그레이션도 `capacity` 행이 발견되면 중단하도록 방어 조건을 두었다.
- 프런트엔드와 DB의 적용 순서가 잠시 어긋나도 기존 제품을 재고 미설정으로 오해하지 않도록 이전 스키마 호환 처리를 유지한다.

## Current Status

- 프런트엔드 코드, 계산 로직, 테스트, Supabase 마이그레이션 작성이 완료됐다.
- 운영 DB와 GitHub에는 아직 적용하거나 푸시하지 않았다.
- 로컬 로그인 화면까지 실행을 확인했으나 테스트 브라우저에 로그인 세션이 없어 제품 추가 화면의 시각 검증은 수행하지 않았다. 운영 데이터는 변경하지 않았다.
- 샌드박스의 Vite 설정 경로 접근 제한으로 일반 검증이 한 차례 막혔지만, 동일 명령을 허용된 작업공간 환경에서 다시 실행해 빌드와 테스트가 모두 통과했다.
