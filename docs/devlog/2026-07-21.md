# 2026-07-21 — 재고 단위 재설계와 Purchase Log 추가

## 1. 개수와 용량 입력 구조 문제 발견

스빈또 수세미를 등록하면서 현재 앱의 개념 혼합이 드러났다.

입력값:

```text
제품 전체 용량: 200g
현재 남은 양: 100
```

기존 구조에서는 현재 재고 단위가 기본값 `개`로 남아 목록에 `100개`로 표시됐다.

이는 사용자가 잘못 입력한 문제가 아니라 앱이 다음 두 정보를 분리하지 못한 문제였다.

- 현재 재고를 무엇으로 세는가
- 제품 전체 용량이 얼마인가

## 2. 재고 입력을 개수형 / 용량형으로 재설계

제품 등록 흐름을 다음처럼 변경했다.

```text
제품명
→ 재고 기준 선택
  - 개수로 관리
  - 용량으로 관리
→ 선택에 맞는 입력창 표시
```

### 개수로 관리

```text
재고 단위: 통 / 팩 / 병
현재 실제 재고: 4
```

### 용량으로 관리

```text
용량 단위: g / ml
새 제품 1개의 전체 용량: 200
현재 남은 총 용량: 100
사용 인원: 1
```

### 관련 수정

- 목록과 카드에 `100g`, `1600ml`처럼 실제 단위 표시
- 입고·재고 정정 다이얼로그를 용량 기준에 맞게 수정
- 현재 남은 용량 비율로 예상 잔여일 계산
- 소진 시 제품 1개분 용량 차감
- 기존 스빈또 수세미 데이터를 `200g 제품 / 현재 100g`으로 변환
- 용량 계산 테스트 추가
- Supabase RPC와 데이터 규칙 수정

### 제품별 주구매처

제품 추가·설정 화면에서 주구매처를 선택할 수 있게 했다.

주구매처는 제품을 구매처별로 묶어보는 기준이고, 개별 구매 기록에는 실제 구매처를 별도로 저장한다.

## 5. 구매 기록 입력

제품 카드에 다음 기능을 추가했다.

```text
구매 기록
과거 기록 한꺼번에
```

구매 기록 필드:

```text
구매일
구매처
구매 수량
제품 1개 용량
총 결제금액 (선택)
배송비 (선택)
메모 (선택)
```

### 핵심 결정: 구매와 입고 분리

```text
구매 기록
→ 언제 어디서 무엇을 샀는지 저장
→ 현재 재고는 변경하지 않음

입고 기록
→ 실제 물건이 도착했을 때 현재 재고 증가
```

과거 쿠팡 기록을 입력해도 현재 구연산 재고가 수십 kg 늘어나는 문제를 방지한다.

## 6. 과거 구매 기록 일괄 입력

동일한 구매처와 규격으로 반복 구매한 제품은 날짜 여러 개를 붙여넣을 수 있게 했다.

지원 예:

```text
2024-02-10
2024. 6. 21.
2024년 11월 3일
2025-03-15
2025-08-02
```

처리 규칙:

- 구매처, 구매 수량, 제품 용량은 공통값으로 한 번 입력
- 중복 날짜 제거
- 잘못된 날짜 제외
- 미래 날짜 제외
- 현재 재고에는 반영하지 않음

## 7. 구매 주기 계산

구매일이 두 개 이상이면 최근 구매 간격의 중앙값을 계산한다.

표시 예:

```text
구매 기록        5회 · 최근 2025년 8월 2일
평소 구매 간격   약 4.4개월
다음 구매 예상   2025년 12월 14일
```

원칙:

- 같은 날 여러 주문은 구매 주기 계산에서 날짜 하나로 취급
- 실제 개봉→소진 주기와 구매 주기를 별도로 표시
- 사용 기록이 부족한 초기에는 과거 구매 간격을 참고 가능

## 8. 구매처별 보기

메인 화면에 보기 전환을 추가했다.

```text
기본 목록
주구매처별
```

구매처별 보기에서는 다음을 표시한다.

- 구매처별 제품 그룹
- 그룹별 제품 수
- 현재 구매 필요 제품 수
- 30일 안 구매 예상 제품 수

목적은 배송비나 최소 주문금액이 애매할 때 같은 구매처에서 곧 사게 될 제품을 빠르게 확인하는 것이다.

이번 단계에서는 제품을 모아 보여주는 것까지만 구현했고, 특정 제품을 앞당겨 함께 사도록 추천하는 기능은 다음 단계로 남겼다.

# 남은 구조적 과제

## 1. 재고 기준과 사용 기록 방식 분리

현재 내부의 `tracking_mode`는 여전히 다음 두 개념을 동시에 맡고 있다.

```text
재고 기준
- count
- capacity

사용 기록 방식
- 수량 차감
- 개봉→소진
```

따라서 현재 필요한 조합 중 하나인 다음 형태는 아직 자연스럽게 지원하지 못한다.

```text
허블룸 크림
현재 재고: 5통
한 통 용량: 50ml
사용 기록: 한 통씩 개봉→소진
```

향후에는 `stock_basis`와 `usage_tracking`을 분리해 `count + open/finish cycle`을 지원하는 것이 좋다.

## 2. 세일 전략 관련 기능

다음 단계로 보류했다.

- 좋은 가격과 긴급 구매 가격
- 할인 유형과 쿠폰
- 배송비 무료 기준
- 같은 구매처에서 곧 살 제품을 앞당겨 사는 추천
- 다음 세일까지 버틸 수 있는지 계산
- 세일 시 추천 구매량
- 계절성 제품의 시즌 필요량

## 3. 통계와 Batch

아직 구현하지 않았다.

- 연간 예상 소비량
- 최근 12개월 실제 소비량
- 시즌별 소비량
- 개별 batch와 유통기한
- 여러 제품 동시 개봉

## 11. 폼 메뉴와 모바일 날짜 입력 폭 보정

### Work Summary

입력창보다 좁게 보이던 사용자 드롭다운 메뉴와 iPhone에서 오른쪽으로 넘치던 날짜 입력창의 공통 폭 문제를 보정했다.

### Work Completed

- 사용자 메뉴 패널 폭을 `260px`에서 반응형 최대 `360px`로 확대
- 제품·구매 폼의 입력 컨트롤에 `min-width: 0`, `max-width: 100%` 적용
- 날짜 입력에 부모 폭 기준의 명시적 폭과 블록 레이아웃 적용
- 모바일 날짜 입력 내부 여백을 조정해 iOS 기본 컨트롤의 폭 초과 가능성 축소

### Decisions

고정 폭을 더 크게 지정하는 대신 `calc(100vw - 24px)`를 함께 사용했다. 작은 iPhone에서는 화면 안쪽 여백을 유지하고, 큰 화면에서는 메뉴가 입력 영역과 지나치게 벌어지지 않도록 하기 위한 결정이다.

### Current Status

CSS 수정이 완료되었고 Vite 빌드와 전체 테스트 9개가 통과했다. 변경 사항은 `codex/fix-mobile-form-widths` 브랜치의 로컬 커밋에 기록했다. 실제 iPhone Safari에서 날짜 입력을 열고 저장하는 동작은 배포 후 최종 확인이 필요하다.

### Next Steps

- 원격 작업 브랜치 푸시

## 12. iPhone 날짜 입력과 드롭다운 높이 재보정

### Work Summary

첫 번째 폭 보정만으로는 iPhone 날짜 입력의 Safari 기본 최소 폭과 드롭다운의 짧은 항목 높이가 해결되지 않아 컨트롤 자체를 다시 조정했다.

### Work Completed

- 사용자 메뉴 계정 영역 여백 확대
- 사용자 메뉴 항목 높이를 `44px`에서 `56px`로 확대하고 글자 크기를 `14px`로 조정
- 제품·구매 폼의 실제 `select` 높이를 `54px`로 통일
- iOS 날짜 입력에 `-webkit-appearance: none`, 명시적 `box-sizing`, 폭·높이 제한 적용
- WebKit 날짜 값 영역을 왼쪽 정렬하고 내부 최소 높이 지정

### Decisions

날짜 입력은 일반 입력의 `width: 100%`만으로 iOS 기본 컨트롤을 완전히 제한하지 못할 수 있어 Safari 전용 외형을 해제했다. 날짜 선택기는 입력창을 탭하는 동작으로 계속 사용할 수 있다.

### Current Status

CSS와 개발 로그를 수정했으며, Vite 빌드와 전체 테스트를 다시 실행할 예정이다. 실제 iPhone Safari에서 날짜 입력이 화면 밖으로 나가지 않는지 최종 확인이 필요하다.

### Next Steps
