# 2026-07-20 Inventory Tracker Development Log

## Work Summary

특정 GitHub Pages 경로에 고정된 JavaScript 자산 URL 문제와 저장소 원본 배포가 완성 빌드를 덮어쓰는 문제를 수정했다. 하나의 빌드 결과물을 여러 하위 경로에서 사용할 수 있게 했고 Pages 배포 방식을 GitHub Actions로 통일했다.

## Work Completed

* Vite의 프로덕션 base를 `/inventory-tracker/` 고정값에서 `./` 상대 경로로 변경했다.
* GitHub Pages 워크플로가 상대 JavaScript 자산 경로와 실제 빌드 파일 존재 여부를 검증하도록 수정했다.
* README의 배포 검증 설명을 현재 동작과 일치하도록 갱신했다.
* `npm run verify`로 프로덕션 빌드와 재고 계산 테스트 4건을 통과했고, 생성된 HTML이 `./assets/...` JavaScript와 CSS를 참조하는 것을 확인했다.
* GitHub Pages 설정의 `build_type`을 `legacy`에서 `workflow`로 변경해 `main` 원본을 배포하던 Jekyll 작업이 Actions 산출물을 덮어쓰지 않게 했다.
* 최신 `main`을 수동 재배포하고 공개 HTML이 원본 `/src/main.tsx` 대신 빌드된 JavaScript를 참조하는지 확인했다. 공개 JavaScript 파일도 HTTP 200과 443,320바이트 응답을 반환했다.

## Decisions

배포 서비스마다 앱이 올라가는 하위 경로가 달라질 수 있으므로 환경별 경로를 추가하는 대신 상대 base를 사용했다. 이 방식은 GitHub Pages의 `/inventory-tracker/`와 생성형 배포 주소 모두에서 현재 문서 위치를 기준으로 자산을 찾는다.

커스텀 Pages 워크플로와 브랜치 기반 Pages 배포를 함께 유지하면 더 늦게 끝난 작업이 공개 사이트를 덮어쓸 수 있다. 빌드와 검증을 포함한 커스텀 워크플로만 배포 권한을 갖도록 Pages 소스를 GitHub Actions 방식으로 고정했다.

## Current Status
